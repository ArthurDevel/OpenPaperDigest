# Fix OpenRouter PDF Fallback for Large Papers

## High-Level Goal

OpenRouter fails to fetch large PDFs (>~6MB) by URL, returning "Provider returned error" (code 400). The current fallback downloads `content.md` from Supabase Storage, but v2 papers don't have `content.md`, so it sends an empty string to the LLM â†’ empty response â†’ error.

**Fix:** When the URL-based PDF call fails, download the PDF ourselves and re-send as a base64 data URI attachment. Also **comment out** (not delete) the dead v1 `content.md` code paths and the "Copy Full Document" button, with explanatory comments so we can bring them back later. The reason for commenting out rather than deleting: these features worked for v1 papers and we want to restore them once we have an affordable alternative to content.md for v2 papers.

## Important Technical Notes

### Base64 data URI format
OpenRouter accepts both URL strings and `data:application/pdf;base64,...` strings in the `file_data` field of a `FilePart`. The base64 approach bypasses OpenRouter's server-side URL fetching, avoiding the timeout on large PDFs. Tested and confirmed working for PDFs up to 22MB (~31M character base64 strings).

### Memory considerations
A 22MB PDF produces a ~30MB base64 string. Combined with the JSON payload, this uses ~66MB in memory (original buffer + base64 string + serialized JSON). `downloadPdfAsBase64` enforces a 50MB max PDF size and validates `Content-Type: application/pdf` to avoid encoding HTML error pages from arxiv rate limiting.

### Error detection for provider failures
OpenRouter returns provider errors inside an HTTP 200 body: `{"error": {"message": "Provider returned error", "code": 400}}`. The existing `chatCompletion` already throws on these (line 118-124 of openrouter.ts), so the catch block in the summary service will trigger correctly. The base64 fallback call does NOT retry on 4xx errors either (same `chatCompletion` behavior) â€” this is acceptable since a 400 on base64 means the content itself is problematic, not a fetch timeout.

### Plugins on fallback call
The `plugins: [{ id: 'file-parser', pdf: { engine: 'native' } }]` should be included on the base64 fallback call as well â€” OpenRouter still needs the plugin to parse the PDF content from the data URI.
