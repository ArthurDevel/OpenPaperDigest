# Consolidate `external_popularity_signals` into `signals` JSONB Column

## High-Level Goal

Replace the `external_popularity_signals` column (stores a JSON array of `ExternalPopularitySignal` objects) with the existing `signals` JSONB column (flat key-value dictionary). This simplifies the data model, removes a redundant column, and unifies all pre-computed scoring signals into a single place.

**Current state:**
- `external_popularity_signals`: `[{"source": "HuggingFace", "values": {"upvotes": 42, "github_stars": 10}, "fetch_info": {...}, "updated_at": "..."}]`
- `signals`: `{"max_author_h_index": 42}`

**Target state:**
- `signals`: `{"max_author_h_index": 42, "hf_upvotes": 42, "hf_github_stars": 10, "sources": ["HuggingFace"]}`
- `external_popularity_signals`: **dropped**

**Signals key convention:** An absent key means "no data available". A present key means "has data". Values are always their natural type (`hf_upvotes: int`, `sources: string[]`, `max_author_h_index: int`). Never write `null` values — omit the key instead.

## Important Technical Notes

### Migration: Aggregating multi-source data per paper

The `external_popularity_signals` column stores a JSON **array** that may contain entries from multiple sources (e.g. a paper from both HuggingFace and AlphaXiv). The migration must aggregate across all array elements per paper using `GROUP BY`:

```sql
-- Per-element migration: each array element contributes its source-specific keys
-- Process each element individually to build source-prefixed fetch_info keys
DO $$
DECLARE
  r RECORD;
BEGIN
  FOR r IN
    SELECT p.id, elem
    FROM papers p,
         jsonb_array_elements(p.external_popularity_signals::jsonb) AS elem
    WHERE p.external_popularity_signals IS NOT NULL
  LOOP
    UPDATE papers
    SET signals = COALESCE(signals, '{}'::jsonb)
      -- Append source to sources array
      || jsonb_build_object('sources',
           COALESCE(signals->'sources', '[]'::jsonb) || to_jsonb((r.elem->>'source')::text))
      -- Add source-prefixed fetch_info
      || CASE
           WHEN r.elem->>'source' IN ('HuggingFace', 'HuggingFaceTrending')
             THEN jsonb_strip_nulls(jsonb_build_object(
               'hf_upvotes', (r.elem->'values'->>'upvotes')::int,
               'hf_github_stars', (r.elem->'values'->>'github_stars')::int,
               'hf_fetch_info', r.elem->'fetch_info'
             ))
           WHEN r.elem->>'source' = 'AlphaXiv'
             THEN jsonb_build_object('alphaxiv_fetch_info', r.elem->'fetch_info')
           WHEN r.elem->>'source' = 'GoogleResearch'
             THEN jsonb_build_object('google_fetch_info', r.elem->'fetch_info')
           WHEN r.elem->>'source' = 'MicrosoftResearch'
             THEN jsonb_build_object('microsoft_fetch_info', r.elem->'fetch_info')
           WHEN r.elem->>'source' = 'DeepMindResearch'
             THEN jsonb_build_object('deepmind_fetch_info', r.elem->'fetch_info')
           WHEN r.elem->>'source' = 'NvidiaResearch'
             THEN jsonb_build_object('nvidia_fetch_info', r.elem->'fetch_info')
           ELSE '{}'::jsonb
         END
    WHERE id = r.id;
  END LOOP;
END $$;
```

Key points:
- Iterates per-element to correctly handle papers with multiple sources
- Each source gets a `<source>_fetch_info` key preserving the original metadata
- `jsonb_strip_nulls` only applied to HF entries (the only ones with nullable values)
- `sources` array built incrementally by appending to existing array
- `COALESCE(signals, '{}') || ...` preserves existing signals (e.g. `max_author_h_index`)

### Two-phase migration for safe deployment

The migration is split into two files to support safe rolling deploys:

1. **Migration 33** (pre-deploy): Copies data into `signals`, updates RPC. Old column remains — the currently deployed code continues to work.
2. **Deploy code changes**: All code now reads from `signals`, ignores `external_popularity_signals`.
3. **Migration 34** (post-deploy): Drops the now-unused `external_popularity_signals` column.

### extractUpvotes simplification

Current implementation handles double-encoded JSON strings (legacy). New version is trivial:

```ts
function extractUpvotes(signals: Record<string, unknown> | null): number | null {
  if (!signals) return null;
  const upvotes = signals.hf_upvotes;
  return typeof upvotes === 'number' ? upvotes : null;
}
```

### Search service scoring: intentional behavior change

`search.service.ts` `computePopularityScore` currently returns **0.5** (neutral) when no signals exist. After this change it returns **0**. This means papers without HF data will score lower in search results. This is intentional — aligning with feed.service behavior where missing data = 0 contribution.

The search service also uses different weights (0.5/0.3/0.2) than the feed service (0.33 each). This refactor only changes the data source, not the scoring formula. Aligning search weights is a separate concern.
