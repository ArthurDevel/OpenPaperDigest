# On-Demand Summary Generation

## High-Level Goal

Scale paper ingestion to 1000+ arxiv papers/day by removing batch summary generation from the processing pipeline. Instead, generate summaries on-demand when a user opens a paper, using a faster/cheaper model (Gemini 3 Flash instead of Gemini 2.5 Pro). Standardize embeddings on title + abstract (instead of title + summary) for consistency across all papers.

**Key decisions:**
- New paper status `partially_completed`: paper has OCR content, thumbnail, embedding — but no summary yet
- Embeddings always use `title + abstract` — never summary — to keep a single consistent strategy
- On-demand summary generation lives in a Next.js API route (not the Python worker)
- Batch summary generation code is preserved but disabled (skipped in pipeline)
- Arxiv abstracts are stored in a new `abstract` column on the papers table
- Non-arXiv papers will have NULL abstract; their embeddings fall back to title-only
- Structured author data with Semantic Scholar stats (h-index, citation count) stored in normalized `authors` + `paper_authors` tables

**Cost impact (benchmarked):**
| | Gemini 2.5 Pro (batch) | Gemini 3 Flash (on-demand) |
|---|---|---|
| Avg time/paper | 29s | 9s |
| Avg cost/paper | $0.05 | $0.011 |
| 1000 papers/day | $50/day | $11/day (only for viewed papers) |

## Important Technical Notes

### Race condition prevention in on-demand generation

Multiple users may open the same unsummarized paper simultaneously. The `generateSummaryForPaper` service handles this with a conditional update:

```typescript
// After generating, atomically write only if summary is still null
// Use a Postgres function to do a conditional JSON merge in one query
const { data, error } = await supabase.rpc('set_paper_summary_if_null', {
  p_paper_uuid: paperUuid,
  p_summary: generatedSummary,
});

if (!data?.was_set) {
  // Another request already wrote a summary — fetch and return it
  const { data: paper } = await supabase
    .from('papers')
    .select('summaries')
    .eq('paper_uuid', paperUuid)
    .single();
  return { summary: paper.summaries.five_minute_summary, alreadyExisted: true };
}
```

The `set_paper_summary_if_null` RPC (added in the migration) performs the check-and-update atomically:

```sql
CREATE OR REPLACE FUNCTION set_paper_summary_if_null(p_paper_uuid TEXT, p_summary TEXT)
RETURNS TABLE(was_set BOOLEAN) AS $$
BEGIN
  UPDATE papers
  SET summaries = COALESCE(summaries, '{}'::jsonb) || jsonb_build_object('five_minute_summary', p_summary),
      status = 'completed'
  WHERE paper_uuid = p_paper_uuid
    AND (summaries IS NULL OR summaries->>'five_minute_summary' IS NULL);

  RETURN QUERY SELECT FOUND;
END;
$$ LANGUAGE plpgsql;
```

This eliminates the TOCTOU race. If two requests generate simultaneously, the first one wins atomically — the second gets `was_set = false` and returns the already-written summary. Cost of a wasted LLM call (~$0.01) is acceptable.

- New papers: `title + abstract`

This is a follow-up task: after running `backfill_abstracts_dag`, re-run `backfill_embeddings_dag` (after setting existing embeddings to NULL) to standardize everything on `title + abstract`. Cost: ~$0.002/paper × 983 papers ≈ $2 one-time.

### Author data via Semantic Scholar

**Linking arxiv → Semantic Scholar:** Call `GET /paper/ARXIV:{arxiv_id}?fields=authors,authors.authorId,authors.name`. This returns each author with their stable `s2_author_id`. From there, fetch full stats per author via `GET /author/{s2_author_id}?fields=paperCount,citationCount,hIndex,affiliations,homepage`.

**Rate limits:** Free tier is 100 requests/5 minutes. With an API key (free to request at semanticscholar.org) it's 1 req/second. At 1000 papers/day with ~4 authors each:
- Paper author lookups: 1000/day (one call per paper using batch endpoint)
- Author stat refreshes: ~4000 unique authors initially, then only new/stale ones
- Well within limits with an API key + caching

**Caching strategy:** The same authors appear across many papers. The `authors` table acts as a cache:
- On ingestion: upsert author by `s2_author_id` — create if new, skip if exists
- Stats refresh: only fetch stats for authors where `stats_updated_at` is NULL or older than 7 days
- This reduces API calls dramatically after initial backfill

**Why not store on the papers table:** Author reputation is paper-independent. An author's h-index applies to all their papers. Normalizing into a separate table avoids duplicating stats across rows and makes it trivial to refresh stats in bulk.

**Existing `papers.authors` column:** Kept as-is for display. It's a denormalized comma-separated string used throughout the frontend. The structured `authors` + `paper_authors` tables are the source of truth for author identity and stats.
