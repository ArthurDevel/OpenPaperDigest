# Move Large Storage Items to Supabase Storage

## High-Level Goal

Migrate large binary/text data out of the PostgreSQL `papers` table and into Supabase Storage. Each paper currently stores a ~2MB JSON blob (`processed_content`) and a base64 thumbnail (`thumbnail_data_url`) directly in database rows. This inflates database size and cost (database disk is 6x more expensive than Supabase Storage per GB). After migration, the database stores only relational metadata and small fields, while all heavy content lives in Supabase Storage under a per-paper folder structure.

**Storage bucket structure:**
```
bucket: papers (public)

{paper_uuid}/
  thumbnail.png              -- first-page thumbnail (currently base64 in DB)
  content.md                 -- final_markdown (raw OCR output)
  sections.json              -- rewritten sections array
  figures.json               -- figure metadata (identifier, short_id, bounding_box, location_page, etc.)
  metadata.json              -- usage_summary, processing stats, cost info
  figures/
    {figure_identifier}.png  -- each extracted figure as a standalone image
```

**What stays in the database:**
- All relational columns (`title`, `authors`, `arxiv_id`, `status`, `num_pages`, etc.)
- `summaries` JSON column (4KB, used for homepage)
- `external_popularity_signals` JSON column

**What moves to storage:**
- `processed_content` column (2MB JSON blob) -> split into the files above
- `thumbnail_data_url` column (base64 PNG) -> `thumbnail.png` file

## Per-File Methods and Types

### worker/papers/storage.py (NEW)

Storage client wrapper for uploading/downloading paper assets to Supabase Storage.

- Initializes Supabase client using SUPABASE_URL and SUPABASE_SERVICE_KEY
- All paths use the pattern `{paper_uuid}/{filename}`

#### Types

```
BUCKET_NAME = "papers"

StoredPaperContent {
  finalMarkdown: str
  sections: list[dict]
  figures: list[dict]        -- figure metadata from figures.json (without image bytes, URLs constructed from paper_uuid + figure_id)
  metadata: dict             -- usage_summary, processing stats
}
```

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `upload_paper_assets` | `paper_uuid: str, thumbnail_bytes: bytes, final_markdown: str, sections: list[dict], figures: list[FigureData], metadata: dict` | `None` | Uploads all paper assets to storage. Encodes thumbnail and figures as PNG, stores markdown as text, sections and metadata as JSON. Uses upsert to allow re-processing. |
| `download_paper_content` | `paper_uuid: str` | `StoredPaperContent` | Downloads content.md, sections.json, figures.json, and metadata.json. Does NOT download figure images (those are served by URL). |
| `download_markdown` | `paper_uuid: str` | `str` | Downloads only content.md |
| `get_thumbnail_url` | `paper_uuid: str` | `str` | Returns public URL for thumbnail.png |
| `get_figure_url` | `paper_uuid: str, figure_id: str` | `str` | Returns public URL for a specific figure image |
| `delete_paper_assets` | `paper_uuid: str` | `None` | Deletes all files under {paper_uuid}/ |

## Important Technical Notes

### Base64 to binary conversion

Figures and thumbnails are currently stored as data URLs (`data:image/png;base64,<data>`). When uploading to storage, strip the prefix and decode:

```python
# "data:image/png;base64,iVBOR..." -> raw PNG bytes
header, b64_data = data_url.split(",", 1)
png_bytes = base64.b64decode(b64_data)
```

The frontend currently renders figures via `<img src={imageDataUrl}>` where `imageDataUrl` is the full data URL. After migration, this becomes `<img src={imageUrl}>` where `imageUrl` is a regular HTTPS URL from Supabase Storage. No other rendering logic changes needed.

### Backfill ordering and safety

The backfill DAG and the column-drop migration must be sequenced carefully:

1. Deploy code with dual-write (Phase 2): new papers write to both storage AND `processed_content`
2. Run backfill DAG (Phase 3) to copy all existing data to storage
3. Verify all papers are accessible from storage
4. Switch read paths to storage-only (Phase 4) -- no fallback logic, throw if storage file is missing
5. Drop the DB columns (Phase 5)

No fallback logic is needed. The backfill runs before the read path is switched, so all data is guaranteed to be in storage before reads are redirected.

### Public bucket considerations

The `papers` bucket should be **public** for reads since:
- Paper content is already publicly accessible via the website
- Public buckets have better CDN caching and performance
- Thumbnails and figures can be served directly without signed URLs
- Upload/delete operations still require authentication (service role key)

### Supabase Storage Python SDK

The worker currently connects to Supabase PostgreSQL directly via SQLAlchemy + psycopg2 (no Supabase SDK). For storage access, add the `supabase` Python package:

```python
from supabase import create_client
client = create_client(SUPABASE_URL, SUPABASE_SERVICE_KEY)
client.storage.from_("papers").upload(path, file_bytes, {"upsert": "true"})
```

This requires adding `supabase` to `worker/requirements.txt`.

- All existing papers are accessible from Supabase Storage (no data loss)
- Paper detail page renders correctly with figures served as URLs (not base64)
- Homepage loads thumbnails from storage public URLs
- Markdown view works from storage
- New papers processed by Airflow are stored in Supabase Storage (not DB)
- `processed_content` and `thumbnail_data_url` columns are dropped from the database
- Database row size per paper drops from ~2MB to ~5KB
- No regressions in page load times (storage URLs should be faster than base64 decoding)
