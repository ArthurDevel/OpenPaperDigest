# Feed Recommendation System

## High-Level Goal

Replace the chronological feed (`ORDER BY finished_at DESC`) with a personalized recommendation system powered by pgvector embeddings. The system captures implicit user signals (expand, save, active reading), builds per-user interest clusters (3-5 embedding vectors), and ranks feed candidates using a weighted blend of recency, popularity, and semantic similarity.

All users (anonymous and authenticated) are tracked via Supabase Auth. Anonymous users are created automatically via `signInAnonymously()` and promoted to permanent users on sign-up (user ID preserved, all data retained).

- Uses `IntersectionObserver` to detect if the expanded summary div is in viewport

- Accumulates active time in a ref (not state, to avoid re-renders)
- Expected reading time = `wordCount / 200 * 60 * 1000` (200 wpm -> ms)

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `getRankedFeed` | `page: number, limit: number, excludePaperUuids: string[], userId: string` | `FeedResponse` | Builds preference vectors for the user, fetches candidates, scores, applies diversity injection, paginates, returns top N with slug lookups. |
| `buildPreferenceVectors` | `userId: string` | `number[][]` | Fetches user's preference clusters from `user_preference_clusters`. Returns weighted embedding vectors. Returns empty array if no clusters exist yet. |
| `fetchCandidates` | `excludePaperUuids: string[], preferenceVectors: number[][], limit: number` | `CandidateRow[]` | Similarity-driven candidate retrieval (see technical notes). |
| `scoreCandidates` | `candidates: CandidateRow[], preferenceVectors: number[][]` | `ScoredPaper[]` | Computes recency, popularity, and similarity scores for each candidate. Score = weighted blend. |
| `injectDiversity` | `rankedPapers: ScoredPaper[], preferenceVectors: number[][]` | `ScoredPaper[]` | Every 5th position, picks the highest-scoring paper from outside the top similarity cluster. For users with 0 preference vectors: no-op. For users with 1 vector: picks lowest similarity. |

**Technical notes on candidate retrieval:**

Candidate retrieval is **similarity-driven**. Recency is a scoring factor, not a filter.

**For users WITH preference clusters:**
- For each cluster, call `match_papers_by_embedding` with the cluster embedding to get top M similar papers from the **entire corpus** (no date filter). HNSW index makes this ~5ms per query.
- M = 50 per cluster. With 3 clusters = up to 150 candidates before deduplication.
- Deduplicate across clusters (same paper may be similar to multiple clusters).
- Score all candidates with the weighted formula.
- A paper from 6 months ago surfaces naturally if its similarity score outweighs its lower recency score.

**For users WITHOUT preference clusters (cold start):**
- No ANN query is possible (no preference vectors).
- Fall back to popularity + recency: fetch top `limit * 3` papers ordered by `finished_at DESC`, score with `popularity + recency` only (similarity = 0).
- As soon as the user expands/reads a few papers, clusters start forming and the next page load switches to similarity-driven retrieval.

**Scoring formula:**
- Score = `SIMILARITY_WEIGHT * similarityScore + POPULARITY_WEIGHT * popularityScore + RECENCY_WEIGHT * recencyScore`
- Weights: similarity=0.5, popularity=0.3, recency=0.2 (constants at top of file)
- Similarity score: pre-computed by `match_papers_by_embedding` (cosine similarity). For cold-start fallback: 0.
- Popularity score: normalized from `external_popularity_signals`. NULL = 0.5 (neutral). HuggingFace = `min(upvotes / 100, 1.0)`. Other sources = 0.5.
- Recency score: exponential decay from `finished_at`, half-life of 7 days.
- For cold-start users (no clusters): effectively `0.6 * popularity + 0.4 * recency` (since similarity=0, the remaining weights are renormalized).

### Postgres function: match_papers_by_embedding (NEW)

Small helper function for Pool 2 ANN retrieval. Called via `supabase.rpc()`.

```sql
CREATE OR REPLACE FUNCTION match_papers_by_embedding(
  query_embedding vector(1536),
  match_count int DEFAULT 20,
  exclude_uuids text[] DEFAULT '{}'
)
RETURNS TABLE (
  paper_uuid text,
  title text,
  authors text,
  thumbnail_data_url text,
  finished_at timestamptz,
  embedding vector(1536),
  external_popularity_signals jsonb,
  similarity float
) AS $$
BEGIN
  RETURN QUERY
  SELECT
    p.paper_uuid, p.title, p.authors, p.thumbnail_data_url,
    p.finished_at, p.embedding, p.external_popularity_signals,
    1.0 - (p.embedding <=> query_embedding) AS similarity
  FROM papers p
  WHERE p.status = 'completed'
    AND p.embedding IS NOT NULL
    AND p.paper_uuid != ALL(exclude_uuids)
  ORDER BY p.embedding <=> query_embedding
  LIMIT match_count;
END;
$$ LANGUAGE plpgsql STABLE;
```

This function uses the HNSW index for fast approximate nearest neighbor search.

### services/interactions.service.ts (NEW)

Persists interactions and updates preference clusters. Cluster updates are async (via Next.js 15 `after()` API).

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `saveInteractions` | `userId: string, events: InteractionEvent[]` | `void` | Batch-inserts interaction events into `user_interactions` table |
| `updatePreferenceClusters` | `userId: string, interactedPaperUuids: string[]` | `void` | Fetches embeddings for interacted papers, assigns each to nearest cluster (or creates new), updates cluster centroid via weighted running average. Called via `after()` in Next.js 15. |
| `getPreferenceClusters` | `userId: string` | `UserPreferenceCluster[]` | Returns user's preference clusters, applying daily weight decay (multiply by 0.95 per day since last update) |

**Technical notes on cluster assignment:**
- For each interacted paper embedding, compute cosine similarity to all existing clusters
- If max similarity > 0.7: update that cluster (`new_centroid = 0.8 * old + 0.2 * paper_embedding`, bump weight and count)
- If max similarity <= 0.7 AND cluster count < 5: create new cluster from paper embedding
- If max similarity <= 0.7 AND cluster count = 5: replace the cluster with lowest weight
- Cluster cosine similarity computation happens in the TypeScript service (max 5 clusters x 1536 dims is trivial)

## Important Technical Notes

### Embedding generation via OpenRouter
Embeddings are generated via the existing OpenRouter integration (`openai/text-embedding-3-small`, 1536 dimensions). No local model, no sentence-transformers, no PyTorch, no Docker changes. The worker already has the OpenRouter API key configured. Cost: ~$0.02 per million tokens (negligible for title + summary).

### Supabase anonymous authentication
All users get a Supabase Auth session -- anonymous users via `signInAnonymously()`, permanent users via magic link. This means:
- All interactions are stored server-side from the start (no localStorage sync)
- All users have preference clusters (no special anonymous logic)
- RLS works for everyone (anonymous users have `auth.uid()`)
- When an anonymous user signs up, their user ID is preserved -- all data stays linked

**Setup required:**
1. Enable anonymous sign-ins in Supabase dashboard: Authentication > Providers > Anonymous Sign-In > Enable
2. Configure rate limiting: Authentication > Rate Limits > set anonymous sign-in limit (default: 30/hour, which is fine)
3. Enable CAPTCHA (recommended): Authentication > Bot & Abuse Protection > enable Cloudflare Turnstile or hCaptcha
4. Add periodic cleanup of stale anonymous users (SQL cron or Airflow DAG):
   ```sql
   DELETE FROM auth.users WHERE is_anonymous IS TRUE AND created_at < now() - interval '90 days';
   ```
5. Enable manual linking in project settings (required for anonymous-to-permanent conversion)

### Similarity-driven candidate retrieval
The feed is **similarity-driven**, not recency-driven. For users with preference clusters, candidates come from ANN similarity search against the entire paper corpus (no date filter). Recency is only a scoring boost, not a filter. This means at 1000 papers/day, the feed surfaces what's relevant to the user, not a random slice of today's papers. For cold-start users (no clusters yet), the feed falls back to popularity + recency until interactions build up clusters.

### Ranking in TypeScript, not SQL
All scoring/ranking happens in TypeScript (`feed.service.ts`), not in a Postgres RPC function. This keeps the ranking logic readable, testable, and easy to iterate on. The only Postgres function is `match_papers_by_embedding` -- a thin wrapper around the pgvector `<=>` operator for ANN search. The rest is standard Supabase `.select()` queries + TypeScript math.

### Next.js 15 upgrade
The project upgrades from Next.js 14.2.30 to 15.x as the first step. This unlocks the `after()` API for running code after response is sent (used for async cluster updates). Key changes:
- React 18 -> 19
- Test for hydration issues after upgrade
- `after()` is available in Route Handlers and Server Actions

### external_popularity_signals encoding
The `external_popularity_signals` column is `JSON` type in Postgres, but the Python worker may double-encode it (JSON string inside JSON). Verify actual DB storage format before implementing the popularity score normalization. If double-encoded, fix in the worker.

### database.types.ts regeneration
After all migrations, run `npx supabase gen types typescript` to regenerate types. The `vector(1536)` column will appear as `string` in generated types (PostgREST represents vectors as string). Serialization between `number[]` and Postgres vector string format is handled in the service layer.

## Success Criteria

1. Every new paper gets a 1536-dim embedding generated via OpenRouter and stored in pgvector
2. All existing completed papers have embeddings after backfill DAG runs
3. All visitors automatically get a Supabase Auth session (anonymous or permanent)
4. User interactions (expand, read, save) are tracked and persisted server-side for all users
5. Active reading time is measured via intersection observer + activity heartbeat, filtering out distractions
6. All users have 1-5 preference clusters that evolve over time
7. Feed is similarity-driven: ANN search against the entire corpus for users with clusters, popularity+recency fallback for cold-start users
8. Feed endpoint returns papers ranked by `0.5*similarity + 0.3*popularity + 0.2*recency`
9. Every 5th paper in the feed comes from outside the user's top similarity cluster (diversity)
10. Cluster weights decay daily (0.95 factor), preventing stale interests from dominating
11. Feed falls back to popularity + recency ordering when no preference data exists (cold-start users)
12. RLS policies protect user interaction and cluster data
13. Anonymous users are promoted to permanent users on sign-up with all data preserved
