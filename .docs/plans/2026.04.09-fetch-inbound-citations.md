# Fetch Inbound Citations DAG

## High-Level Goal

Add a new Airflow DAG (`fetch_inbound_citations`) that fetches "papers that cite our papers" from the Semantic Scholar citations batch API. This enriches the citation graph with inbound edges -- currently we only have outbound references from our papers.

The key constraint is **preventing exponential graph growth**. A strict two-hop boundary must be enforced:

1. **Hop 0 (our papers)**: Outbound references fetched by `citation_graph` DAG. Inbound citations fetched by the **new** `fetch_inbound_citations` DAG.
2. **Hop 1 (external nodes)**: Outbound references fetched by `expand_external_references` DAG. No inbound citations fetched.
3. **Hop 2+**: Nothing fetched. The graph stops growing.

**Design decision: fetch-once.** Inbound citations are fetched once per paper. Once a paper has its inbound sentinel, it is not re-fetched in future runs. This mirrors the outbound reference behavior. New citations that appear after the initial fetch are not picked up.

### Audit of `expand_external_references` DAG

The existing `expand_external_references` DAG fetches outbound references for any `cited_s2_id` that has never appeared as a `source_s2_id`. When the new inbound citations DAG introduces new citing papers as `source_s2_id` entries, the question is whether this creates runaway growth.

**This is safe because:**
- The expand DAG only fetches **outbound references** (not inbound citations) for external nodes.
- Once an external node's outbound references are fetched, it gets a `source_s2_id` entry (real edges or sentinel), so it won't be fetched again.
- Citing papers introduced by inbound edges appear as `source_s2_id`, NOT as `cited_s2_id`. The expand DAG's query (`fetch_external_nodes_needing_references`) finds nodes that appear as `cited_s2_id` but NOT as `source_s2_id`. So citing-only papers are **not** eligible for expansion.
- A citing paper X only gets expanded if X also appears as `cited_s2_id` somewhere (i.e., some other paper references X). This happens naturally when our papers or expanded external nodes cite X.
- The inbound citations DAG **only queries papers in the `papers` table**, not external nodes. This is the critical boundary that prevents exponential growth.

**Growth analysis:** If we ingest N papers/day and each has ~K inbound citations, the inbound DAG adds ~N*K new source nodes per week. The expand DAG only processes the subset of these that also appear as `cited_s2_id` targets, which is a fraction. Growth is linear in N, not exponential.

## Important Technical Notes

### Tracking inbound fetch completion: nullable `source_s2_id` sentinel

The `paper_citations` table stores edges as `(source_s2_id, cited_s2_id)`. Regular edges have both fields populated regardless of whether they came from an outbound reference fetch or inbound citation fetch -- there is no need to distinguish direction on regular edges (the existing unique index `uq_paper_citations_edge` on `(source_s2_id, cited_s2_id) WHERE cited_s2_id IS NOT NULL` handles dedup).

The only question is how to track "has this paper's inbound citations been fetched?" to avoid re-fetching. The outbound convention uses a sentinel row `(source=our_paper, cited=NULL)`. The natural inbound analog is `(source=NULL, cited=our_paper)` -- symmetric and self-documenting.

This requires making `source_s2_id` nullable (currently NOT NULL). The migration:
- `ALTER TABLE paper_citations ALTER COLUMN source_s2_id DROP NOT NULL`
- Add unique index: `CREATE UNIQUE INDEX uq_paper_citations_inbound_sentinel ON paper_citations (cited_s2_id) WHERE source_s2_id IS NULL`

The existing outbound sentinel index (`uq_paper_citations_sentinel` on `source_s2_id WHERE cited_s2_id IS NULL`) is unaffected. The existing edge index (`uq_paper_citations_edge` on `source_s2_id, cited_s2_id WHERE cited_s2_id IS NOT NULL`) is unaffected since regular edges always have both fields populated.

### Queries that must be updated for nullable `source_s2_id`

Making `source_s2_id` nullable introduces inbound sentinel rows that match some existing WHERE clauses. Three queries must be updated to add `AND source_s2_id IS NOT NULL`:

1. **`citation_graph_dag.py` `compute_scores` task** (line 343): Edge-loading query `SELECT source_s2_id, cited_s2_id FROM paper_citations WHERE cited_s2_id IS NOT NULL`. Without the fix, inbound sentinels `(source=NULL, cited=our_paper)` would be returned as edges, creating a NULL node in the networkx graph.

2. **`citation_graph_dag.py` `write_paper_scores` cited_by_count query** (line 178): `SELECT cited_s2_id, COUNT(*) FROM paper_citations WHERE cited_s2_id IS NOT NULL GROUP BY cited_s2_id`. Without the fix, inbound sentinels would inflate cited_by_count by 1 for every paper with fetched inbound citations.

3. **`expand_external_references_dag.py` `fetch_external_nodes_needing_references` query** (line 41): The `FROM paper_citations cited` scan should exclude sentinels. Add `AND cited.source_s2_id IS NOT NULL` to the WHERE clause for correctness.

### S2 Citations Batch API

The S2 batch endpoint `POST /paper/batch` with `fields=citations.paperId` returns citations (papers that cite the queried papers). The batch endpoint truncates at **1000 citations per paper** for nested fields. For highly-cited papers, this means we get a partial list. This is acceptable -- the goal is to enrich the graph, not to be exhaustive. There is no way to detect truncation from the response.

The response structure mirrors references:

```json
[
  { "citations": [{"paperId": "abc123"}, {"paperId": "def456"}] },
  null,
  { "citations": [{"paperId": null}] }
]
```

Note: `isInfluential` is not available via the batch endpoint for nested citation/reference fields (same limitation as the existing `fetch_paper_references_batch`). The `S2PaperCitation.is_influential` field will always be `None` from batch calls.

### Preventing re-expansion of inbound-only nodes

When inbound citations are stored, the citing papers become `source_s2_id` entries. The expand DAG's `fetch_external_nodes_needing_references` query finds nodes that appear as `cited_s2_id` but NOT as `source_s2_id`. Citing papers that appear ONLY as `source_s2_id` (from inbound edges) are NOT eligible for expansion -- they need to also appear as `cited_s2_id` somewhere to be picked up. This means:

- If paper X cites our paper P, the edge `(source=X, cited=P)` is stored. X appears as `source_s2_id` only. The expand DAG does NOT pick up X.
- X only gets expanded if some other paper also cites X (X appears as `cited_s2_id`). This happens naturally when our papers or expanded external nodes have X in their references.
- This naturally limits growth without any additional logic.

## Per-File Methods and Types/DTOs

### models.py -- semantic_scholar (MODIFIED)

#### Types

```
S2PaperCitation {
  cited_s2_id: str           # The paper that was queried (our paper, the one being cited)
  citing_s2_id: str          # The paper that cites it (the citing paper)
  is_influential: Optional[bool]  # Always None from batch API
}
```

### models.py -- papers/db (MODIFIED)

- `PaperCitationRecord.source_s2_id`: change from `nullable=False` to `nullable=True`

### client.py (MODIFIED)

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `fetch_paper_citations_batch` | `s2_paper_ids: List[str]` | `List[S2PaperCitation]` | Batch fetch citations (papers that cite these papers) from S2. 500 per request. Skips null entries and null citingPaper.paperId. Uses `fields=citations.paperId`. |

### citation_helpers.py (MODIFIED)

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `fetch_and_store_inbound_citations` | `session: Session, s2_paper_ids: List[str]` | `Dict[str, int]` | Batch-fetch inbound citations from S2 in chunks of `BATCH_SIZE` (500). INSERT edges as `(citing_id, our_id)` with ON CONFLICT DO NOTHING. Papers with 0 citers get an inbound sentinel row `(source=NULL, cited=our_id)`. Returns counts dict with `edges_inserted`, `sentinel_rows`, `errors`, `batches_processed`. |

### citation_graph_dag.py (MODIFIED)

No new methods. Two SQL queries updated to add `AND source_s2_id IS NOT NULL`:
- `compute_scores` edge-loading query
- `write_paper_scores` cited_by_count query

### expand_external_references_dag.py (MODIFIED)

No new methods. One SQL query updated to add `AND cited.source_s2_id IS NOT NULL` in `fetch_external_nodes_needing_references`.

### fetch_inbound_citations_dag.py (NEW)

DAG that runs weekly on Tuesday at 08:00 UTC. Single task: find papers needing inbound citation fetch, call `fetch_and_store_inbound_citations`.

#### Methods

| Method | Arguments | Returns | Description |
|--------|-----------|---------|-------------|
| `fetch_papers_needing_inbound_citations` | `session: Session` | `List[str]` | Query papers with `s2_paper_id` that don't yet have an inbound sentinel (`source_s2_id IS NULL AND cited_s2_id = s2_paper_id`). Only queries the `papers` table -- never external nodes. |

### Migration (NEW)

#### Changes

- Make `source_s2_id` nullable on `paper_citations`
- Add unique index `uq_paper_citations_inbound_sentinel` on `(cited_s2_id) WHERE source_s2_id IS NULL`
