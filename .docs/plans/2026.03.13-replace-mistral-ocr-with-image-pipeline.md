# Replace Mistral OCR with PDF-Direct-to-LLM Pipeline

## High-Level Goal

Replace the Mistral OCR step ($30/day) in the paper processing pipeline by sending the PDF directly to Gemini 3 Flash via OpenRouter's `file` content type. Instead of PDF → Mistral OCR → markdown → LLM summary, we do PDF URL → Gemini 3 Flash → summary.

**Cost savings**: eliminates ~$900/month in Mistral OCR costs. Test results on a 144-page paper showed PDF-via-URL costs $0.0066/paper (vs $30/day OCR). At 1000 papers/day ≈ $6.60/day.

**Test results** (arxiv 2303.18223, 144 pages, 5.6 MB):

| Method | Cost | Time | Prompt tokens |
|--------|------|------|---------------|
| PDF base64 via `file` type | $0.0398 | 29.1s | 75,439 |
| PDF URL via `file` type | **$0.0066** | 36.4s | 75,439 |

Both produced high-quality summaries. PDF URL is 6x cheaper — use this approach.

**What we lose (accepted trade-offs)**:
- No more extracted figures with bounding boxes (to be revisited later)
- No more `final_markdown` (OCR text) stored per paper
- No more inline image references in markdown

**What stays the same**:
- Metadata extraction (already uses page images from first 3 pages)
- Embedding generation (uses title + abstract)
- Abstract summary generation (uses abstract text)
- Thumbnail generation (uses first page image)

**Behavioral change**: Re-enabling `generate_five_minute_summary` in the worker means new papers will get `status=completed` (not `partially_completed`). The on-demand web summary path will only be used for re-summarization of old papers.

## Important Technical Notes

### OpenRouter file content type format
The correct format for sending PDFs via OpenRouter:
```json
{
  "type": "file",
  "file": {
    "filename": "paper.pdf",
    "file_data": "https://arxiv.org/pdf/2303.18223"
  }
}
```
- Field name is `file_data` (not `data` or `url`)
- Accepts either a URL string or a `data:application/pdf;base64,...` data URI
- URL is 6x cheaper than base64 for the same token count — always prefer URLs
- Must include `plugins: [{"id": "file-parser", "pdf": {"engine": "native"}}]` in the request to avoid fallback to mistral-ocr ($2/1000 pages)

### PDF URL sources
- **Worker pipeline**: Papers come from arxiv — use the arxiv PDF URL directly (e.g., `https://arxiv.org/pdf/2303.18223`)
- **On-demand web summary**: PDF is stored in Supabase Storage — generate a signed URL for `{uuid}/paper.pdf`
- **Non-arxiv papers**: Upload PDF to Supabase first, then use signed URL

### Storage schema change
Old: `{uuid}/content.md`, `{uuid}/figures/{id}.png`, `{uuid}/figures.json`, `{uuid}/sections.json`
New: `{uuid}/paper.pdf`, `{uuid}/metadata.json`, `{uuid}/thumbnail.png`

Existing papers in storage keep their old format. Use `pipeline_version` field in `metadata.json` to distinguish formats: version 2 = PDF pipeline, absent/version 1 = old OCR format.

### Backwards compatibility for existing papers
~All existing papers already have summaries. The on-demand summary service only runs for papers with `status=partially_completed`. New papers will use the PDF pipeline. For any edge case where an old paper needs re-summarization, the text fallback in the web service handles it.

### Abstract summary fallback
`generate_abstract_summary` currently falls back to first 1000 words of `final_markdown` if no abstract is available. Since `final_markdown` will no longer exist, this fallback must change. Replace the `raise RuntimeError` with `logger.warning` + `return` (skip abstract summary gracefully). Most arxiv papers have abstracts from arxiv metadata.

### No more page image conversion for summaries
The previous plan required converting all PDF pages to images. With the PDF URL approach, this is no longer needed. `pdf_to_image.py` still converts the first 3 pages for metadata extraction and thumbnail — no changes needed there.

### Memory and payload benefits
- No need to hold 70+ page images in memory
- No need to upload page images to Supabase Storage (~50-150KB per page saved)
- Request payload is tiny (just a URL string vs megabytes of base64 images)
- Supabase storage per paper drops from ~10MB (page images) to ~5MB (just the PDF)

## Success Criteria

1. New papers process without calling Mistral OCR API
2. Summaries generated from PDF URL are comparable quality to OCR-based summaries
3. Mistral OCR daily spend drops to $0
4. On-demand summary generation works for both new papers (PDF URL) and old papers (text fallback)
5. Original PDF stored in Supabase Storage and retrievable via signed URL
6. No regression in metadata extraction, embedding generation, or abstract summaries
7. Processing cost per paper < $0.01 (confirmed: $0.0066 for 144-page paper)
