# Hide Previously Seen Papers from Feed

## High-Level Goal

Papers a user has scrolled past in previous sessions should not appear again in their feed. Today, the feed's `excludePaperUuids` mechanism only excludes papers loaded in the **current session** (from in-memory state). We need to:

1. Persist "seen" impressions to the `user_interactions` table (currently they only go to Umami analytics)
2. Query previously-seen paper UUIDs server-side in `getRankedFeed` and merge them into the exclude list

## Important Technical Notes

### Deduplication of `seen` events in `trackSeen`

`usePaperImpression` already deduplicates per component instance via `hasBeenSeen` ref, so it fires once per mount. However, as the user scrolls down and back up, React may remount cards. `trackSeen` should check `sessionPaperUuidsRef` before pushing to the buffer to avoid sending duplicate `seen` events within one session.

### `seen` events should NOT trigger preference cluster updates

In the interactions API route (`/api/users/me/interactions/route.ts`), the `after()` callback calls `updatePreferenceClusters` with all paper UUIDs from the batch. Seen events should be filtered out before that call — scrolling past a paper does not signal interest the way expanding or reading does. Filter the `paperUuids` array passed to `updatePreferenceClusters` to only include UUIDs from `expanded` or `read` events. The API route does not validate `interactionType` values (it passes them straight to `saveInteractions`), so no additional validation changes are needed — only the cluster filtering.

### Wiring `onSeen` in `usePaperImpression`

The `onSeen` callback should be called directly inside the `hasBeenSeen` flip block in the IntersectionObserver callback (alongside `trackImpression()`). Since `trackSeen` from `useInteractionTracker` is a stable `useCallback` with no dependencies, it is safe to add `onSeen` to the `useEffect` dependency array without causing unnecessary observer re-creation.

### Volume considerations

A user scrolling through 40 papers generates 40 `seen` rows per session. These are small rows (~100 bytes each) and the existing 15-second flush batching keeps request count low. The `getInteractedPaperUuids` query uses the existing `(user_id, paper_uuid)` composite index and returns only UUIDs — no performance concern at current scale.

### Client-side exclude list becomes redundant

After this change, the server independently queries seen/interacted UUIDs. The client still sends `excludePaperUuids` from `getSessionPaperUuids()` in the POST body — this is redundant but harmless (defense-in-depth). No change needed to the client-side exclude logic.

- [x] In `/api/users/me/interactions/route.ts`, filter `paperUuids` passed to `updatePreferenceClusters`: build `clusterPaperUuids` from events where `interactionType` is `'expanded'` or `'read'` only, and pass that to `updatePreferenceClusters` instead of the full `paperUuids` list. If `clusterPaperUuids` is empty, skip the `after()` call entirely.

> **Note:** This must land before or alongside Phase 3, otherwise `seen` events will pollute preference clusters in the window between deployment.

## Success Criteria

- A user scrolls past 20 papers, closes the tab, and returns later → none of those 20 papers appear in the new feed
- Papers that were expanded, read, or saved in previous sessions are also excluded
- `seen` events do not influence preference clusters (no cluster drift from passive scrolling)
- The feed still works for unauthenticated / cold-start users (no `getInteractedPaperUuids` call when `userId` is null)
- No duplicate `seen` rows are created within a single session for the same paper
